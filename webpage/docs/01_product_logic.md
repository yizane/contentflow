# 13 — 产品逻辑结构（UI 设计交接文档）

> 给 UI 设计师：这份文档描述 Flyfus 内容引擎的业务逻辑、工作流和数据结构。
> 目标是设计一个「内容工厂监控台」——用户每天来看：今天生产了什么、卡在哪、哪些要我处理。

---

## 一句话定位

这是一个**自动写文章的流水线工厂**：每天自动从全网抓取行业资讯 → 挑选题 → AI 写文章 → 机器审稿（质量+事实）→ 改写成公众号/抖音/小红书版本 → 打包等人工终审。

用户（运营人员）每天只做三件事：**点一下开工 → 看流水线跑完 → 终审通过的文章**。

---

## 核心工作流（7 步流水线）

这是产品的主轴，UI 必须让用户一眼看到「今天的流水线跑到哪了、每步产出了什么」。

| 步骤 | 名称 | 做什么 | 产出（用户关心的结果） | 可能状态 |
|---|---|---|---|---|
| 1 | **采集选题源** | 抓取 42 个信息源（亚马逊官方/行业媒体/中文跨境/搜索引擎） | 「今天抓到 222 条资讯」+ 哪些源失败了（403/超时） | 成功/有警告/失败 |
| 2 | **生成主题池** | AI 从资讯里提炼 10-30 个候选选题并打分，自动去重（避免和最近写过的重复） | 「新增 20 个候选选题，1 个因重复被拒」+ 选题列表（标题/分数/优先级） | 成功/失败 |
| 3 | **创建文章任务** | 从主题池挑出最高分的选题立项 | 「选中：xxx 选题（92 分）」 | 成功/无可选（分数都不够） |
| 4 | **生成文章** | AI 写全文（5000+ 字，含 FAQ/表格/来源），自带质量自检，不合格自动重写（最多 3 次） | 一篇文章 + 质量分（0-100）| 成功/失败 |
| 5 | **事实核查** | 另一个 AI 当审稿人，逐条核查文章里的事实表述，标出需要补来源的地方 | 核查结论：**微调后可发** 或 **待补来源**（+待修清单） | 通过/待补来源/未过 |
| 6 | **渠道改写** | 把文章改写成 3 个版本：公众号长文、抖音口播稿、小红书笔记 | 3 篇渠道稿 | 每渠道独立成功/失败 |
| 7 | **SEO/GEO 评分** | 双维度打分：SEO（搜索引擎收录）+ GEO（被 ChatGPT/AI 引用的就绪度） | SEO 分 / GEO 分 / 综合分 | 已评分/跳过（文章未就绪不评） |

**额外的修复回路（不在每日流水线里，按需触发）**：
- **补来源修订**（fix:sources）：第 5 步说「待补来源」的文章，AI 自动去搜官方资料 → 修订文章 → 重新核查。一篇文章可能修 1-3 轮。每轮产生一个新版本（v1→v2→v3）。

---

## 核心对象与生命周期

### 1. 文章（最重要的对象）

文章有**状态机**，UI 上应该用一条进度条/徽章体现：

```
已过质量门 → 待补来源 ⟲(自动修订循环) → 待终审 → 已复审 → 批准发布 → 已发布
                                          ↘ 不通过/归档
```

| 状态 | 含义 | 用户要做什么 |
|---|---|---|
| 待补来源 | 事实核查发现有表述缺权威来源 | 一般不用管（系统自动修）；连续修不动时看「待修清单」人工处理 |
| **待终审** | 机器全部通过，等人看 | **核心动作**：读文章 → 点「通过」或「打回」 |
| 批准发布 | 人工通过 | 等 Web 发布系统取走 |

每篇文章挂着的子数据（详情页的信息架构）：
- **多个版本**（v1 初稿、v2/v3 修订版）——正文都在版本里
- **质量报告**（100 分制 + 7 个维度明细 + 问题清单）
- **事实核查历史**（每轮：N 条表述 / 高风险几条 / 待修几条 + 待修清单文字）
- **来源补全记录**（每条待修表述：找到了什么权威来源、建议怎么改写）
- **3 个渠道稿**（各自的标题、正文、字数）
- **SEO/GEO 评分**（两套细分维度：SEO 9 项 / GEO 8 项，各项满分不同）
- **状态变化历史**（什么时间、从什么状态到什么状态、为什么）
- **AI 调用记录**（内部数据：每次调用的耗时/状态/错误——给排查用，普通视图收起）

### 2. 运行（Run）

一次完整的流水线执行。**一天只允许一次有效的"每日运行"**（防止重复点按钮产生重复文章）。

- 状态：运行中 / 成功 / 部分成功 / 失败 / 已被替代（重建时旧的被替代）
- 控制动作：
  - **跑今天**：当天没跑过才可点
  - **重试失败**：只有失败/部分成功时可点（只补失败的步骤，不重复花钱）
  - **重建今天**：归档今天的旧数据完整重跑（要二次确认）
- 每次点击（包括被拒绝的）都有操作记录（谁、何时、来自页面还是命令行、为何被拒）

### 3. 采集源

42 个信息源，每次运行都有逐源的抓取结果：抓到几条、入库几条（去重后）、HTTP 状态、耗时、失败原因。运营关心：**哪些源持续失败**（需要换源或修反爬）。

### 4. 选题池

候选选题列表：标题、分数（0-100）、优先级（P0/P1/P2）、状态（候选/已选中/已生成/被拒-含拒绝原因如"与最近文章重复"）。

---

## 建议的页面结构（供设计参考，不是强制）

1. **首页 / 今日看板**：今天的流水线（7 步可视化 + 每步结果数字）、待我处理（待终审 N 篇 / 卡住的文章 N 篇）、运行控制按钮
2. **文章库**：列表（状态筛选）→ 详情（正文 HTML 渲染 + 右侧/tab：版本、核查、渠道稿、评分、历史）
3. **运行历史**：每次运行一行 → 详情（流水线步骤 + 采集明细 + 失败的 AI 调用）
4. **选题池**：候选列表 + 被拒原因（让运营理解"为什么今天选了这个题"）
5. **配置管理**（未来 Web 版核心）：关键词库、采集源、提示词/策略文档（都在数据库里，可在线编辑）

## 设计要点

- **结果优先于过程**：每步展示"产出了什么"（数字+示例），过程日志收起
- **状态颜色系统**：绿=成功/通过，黄=警告/待处理，红=失败/拒绝，蓝=进行中，灰=跳过/归档
- **待办驱动**：用户登录第一眼应该看到「今天有 N 件事等你」而不是数据库表
- **内部数据隔离**：AI 提示词全文、模型原始回复不给普通用户看（只显示字数/耗时摘要）

## 数据接口

全部数据在 MySQL（21+3 张表），现有本地 API 见 scripts/view_server.js（13 个端点），表结构与字段契约见 docs/05、docs/09。正式 Web 直接连库即可，无需经过本工具。
