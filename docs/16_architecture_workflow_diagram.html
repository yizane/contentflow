ContentFlow v1.0-rc1 本地运行流水线图

本文档重点画当前项目真实怎么跑：本地通过 Node 启动一个个脚本，脚本从 MySQL 读取上一步结果，必要时内部调用 OpenClaw 完成搜索、生成、核查或评分，然后把结构化结果写回 MySQL，下一步继续读取。

结论先放前面：线上部署时应由业务系统自己的定时器或队列触发 engine:daily 或运行接口，OpenClaw 只做执行器，不建议把每日内容生产托管到 OpenClaw 内部定时任务。

1. 本地运行工业流图

Node 本地脚本 脚本内部调用 OpenClaw 纯规则处理 MySQL 中间结果 补源/模型调用等高风险信息

主链路 1-8 做了什么

1采集来源
做什么从已启用的数据源抓取当天材料。
怎么做RSS/API/网页规则采集；搜索类源才调用 OpenClaw。
写入source_items、采集日志、内容分类。
用途给主题生成提供可追溯素材。
2生成主题池
做什么把来源材料提炼成候选主题。
怎么做把 source、关键词、近期已写主题组装给 OpenClaw。
写入topic_candidates、model_runs。
把关校验 JSON、主题分类、来源 URL、重复和关键词节流。
3创建文章任务
做什么从候选主题里选出今天要写的题。
怎么做按分数、来源支撑、分类配额、重复风险做组合选择。
写入article_jobs 和状态流转记录。
结果入选主题变成待执行写作任务。
4生成文章
做什么根据文章任务生成正文和结构化文章 JSON。
怎么做OpenClaw 按策略写稿，本地校验字段和质量报告。
写入articles、article_versions、quality_reports。
失败模型输出不合格会重试；仍失败则任务标记失败。
5事实核查
做什么检查文章里的事实、数字、结论和来源支撑。
怎么做OpenClaw 输出风险等级、缺来源点和必须修复项。
写入fact_checks、source_resolutions。
结果文章进入待审、待补来源或事实核查失败。
6补来源与修订
做什么修复事实核查指出的缺来源和高风险表达。
怎么做补权威来源、改写局部内容，再重新核查。
写入新版 article_versions 和补源记录。
边界多轮仍不收敛时转人工，不硬推通过。
7渠道改写
做什么把已就绪文章改写成渠道稿。
怎么做按公众号、抖音、小红书的结构分别生成。
写入channel_outputs。
把关检查渠道结构、禁词和必要字段。
8SEO/GEO 评分
做什么给文章做搜索和 AI 引用友好度评估。
怎么做OpenClaw 输出 SEO、GEO、事实分、业务契合度和可读性。
写入seo_geo_scores 和文章分数字段。
用途辅助终审和优化方向，不覆盖文章质量主门禁。

2. 单个 OpenClaw 节点内部怎么执行

注意：当前项目的运行形态就是本地 Node 进程串行启动脚本。OpenClaw 不保存状态，只在某些脚本内部被调用；真正的状态、正文、评分、核查结果和追踪记录都写入 MySQL。

3. 流水线节点执行方式

节点 入口/脚本 执行类型 执行者 主要输入 主要落库/输出 说明
触发每日运行 npm run engine:daily 或本地查看器运行接口 规则 命令行 / 本地查看器 / 外部定时器 mode、daily_key、actor run_actions 只负责发起请求；是否允许由运行控制逻辑判断。
每日幂等控制 engine_daily.js + run_control_lib.js 规则 Node 进程 engine_runs.daily_key、is_active engine_runs、run_actions start（首次运行）、retry（重试）、rebuild（重建）、force（强制）四模式；rebuild 归档不物理删除。
主编排 engine_batch.js 规则 Node 进程 limit（数量）、min-score（最低分）、strategy（写作策略） workflow_steps、workflow_events 逐步创建流水线步骤记录，再以子进程执行每个脚本。
采集来源 pipeline/sources_collect.js 规则 + 智能体 Node 网页采集 + OpenClaw 搜索采集 config_sources、sources_yaml source_items（含分类字段）、source_collection_logs、content_classifications RSS 和网页走规则采集；search_query（搜索查询）走 OpenClaw 搜索；采集后自动做内容分类。
生成主题池 pipeline/topics_generate.js 模型 OpenClaw 智能体 最近 source_items、keywords topic_candidates、model_runs 输出 JSON（含 contentType/businessCategory/topicCluster），经结构约束和业务规则校验，再做去重和关键词节流。
创建文章任务 pipeline/jobs_create.js 规则 Node 进程 候选主题、分数、策略 article_jobs、status_transitions 选择高分候选，不需要调用模型。
生成文章 pipeline/jobs_run.js 模型 OpenClaw 智能体 article_job、prompt、strategy articles、article_versions、quality_reports、model_runs 最多重试；正文和 quality JSON 必须通过本地校验。
事实核查 pipeline/factcheck_run.js 模型 OpenClaw 智能体 文章正文、事实核查提示词 fact_checks、source_resolutions、文章状态 决定 ready_for_review、needs_fact_sources 或 fact_check_failed。
来源补全与修订 pipeline/sources_fix.js、pipeline/sources_resolve.js、pipeline/articles_revise.js 智能体 OpenClaw 智能体 + Node 控制 必须修复的事实点、internal_claims、来源策略 新版 article_versions、source_resolutions、再次事实核查 这是闭环修复节点；多轮仍不收敛时转人工。
渠道改写 pipeline/channels_generate.js 模型 OpenClaw 智能体 已就绪或已校验的文章版本 channel_outputs 生成公众号、抖音、小红书三渠道稿，并做禁词/结构校验。
SEO/GEO 双评分 pipeline/score_seo_geo.js 模型 OpenClaw 智能体 文章正文、strategy seo_geo_scores、文章分数字段、版本 JSON 输出总分、SEO、GEO、事实分、业务契合度、可读性。
发布包 pipeline/package_export.js 规则 Node 进程 文章、版本、渠道稿、质量报告 publish_packages 不真实发布，只生成可交付资产。
人工终审 pipeline/review_mark.js 或本地查看器/网页后台终审接口 人工 运营/审核人 终审状态、备注、操作者 review_actions、status_transitions approved_for_publish/published 不应被自动归档。
内容分类回填 pipeline/content_classify.js 规则 + 模型 Node 规则 + OpenClaw 智能体 未分类的 source_items / topic_candidates / articles、content_taxonomy 实体分类字段、content_classifications、model_runs 规则置信度 ≥ 0.85 直接采用；否则批量调用 OpenClaw；历史数据分批回填。
运行报告 engine_report.js 规则 Node 进程 运行记录、文章、追踪记录、积压状态 engine_reports 用于试运行日报和运维排查；含三层内容分类统计与未分类积压提示。

5. 线上产品形态建议

推荐形态：业务系统调度流水线

线上应把 ContentFlow 做成一个有控制面的内容生产服务，而不是把每日任务放进 OpenClaw 的内部定时任务。推荐组件是：

组件 职责
网页管理后台 看今日状态，触发 start（首次运行）、retry（重试）、rebuild（重建），处理人工终审。
业务定时器 每天固定时间调用运行接口或执行 npm run engine:daily。
运行控制接口 复用 daily_key 幂等逻辑，记录 run_actions。
引擎执行器 执行 engine_batch 或后续队列任务。
OpenClaw 只执行模型/智能体节点，不保存业务状态。
MySQL RDS 唯一运行态数据源，供网页后台和引擎共享。

不建议：OpenClaw 内部定时任务直接跑

OpenClaw 内部定时任务看起来省事，但会把业务控制权放错地方。内容生产需要知道 daily_key、是否已有活跃运行、哪些文章被批准、重建时怎么归档、触发人是谁、失败如何审计。这些都属于业务系统职责。

推荐答案：线上用外部定时器或云厂商定时器触发 ContentFlow 自己的运行接口或命令行；OpenClaw 继续作为执行层。定时器只负责“到点敲门”，是否真的开跑由 run_control_lib 决定。

最小线上部署路径

阶段 形态 触发方式 适用场景
试运行 单台服务器 / 本机常驻代码 + RDS 系统定时任务执行 npm run engine:daily 每天 1-3 篇，人工盯运行结果。
小规模生产 容器化引擎 + 网页管理后台 + RDS 云定时器或 Kubernetes 定时任务调用运行接口 需要可视化、报警、重试、多人查看。
多客户生产 接口控制面 + 队列 + 执行器池 + RDS 定时器只入队，执行器消费任务 需要并发水位、隔离、取消、崩溃恢复。
