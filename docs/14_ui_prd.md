# 14 — Flyfus 内容工厂监控台 UI PRD

> 交付对象：UI/UX 设计师、前端产品负责人。
> 目标：把 Flyfus 内容引擎设计成一个运营每天可用的「内容工厂监控台」。
> 核心体验：用户每天打开系统后，只需要知道今天生产了什么、卡在哪里、哪些文章需要人工处理。

---

## 1. 产品定位

Flyfus 内容工厂监控台是一个面向内容运营人员的生产管理后台。系统每天从行业资讯源采集信息，生成选题，创建文章任务，调用 AI 写文章，完成质量门、事实核查、渠道改写和 SEO/GEO 评分，最后把机器通过的文章交给人工终审。

产品不是数据库管理工具，也不是 AI 调试工具。普通用户第一眼看到的应该是结果和待办，而不是日志、表名、prompt 或模型原始回复。

### 一句话目标

让运营人员每天完成三件事：

1. 点一下「跑今天」。
2. 看流水线是否跑完、哪里异常。
3. 终审通过可以发布的文章。

### 主要用户

| 用户 | 关注点 | 常用功能 |
|---|---|---|
| 内容运营 | 今天产出了什么，哪些文章等我审 | 今日看板、待终审、文章详情、渠道稿 |
| 运营负责人 | 产能是否稳定，失败是否可恢复 | 今日看板、运行历史、趋势摘要 |
| 技术/维护人员 | 哪个源失败、哪个模型调用失败 | 运行详情、采集源明细、Debug 折叠区 |

---

## 2. 设计原则

### 2.1 结果优先

流水线每一步优先展示用户关心的结果，例如「抓到 222 条资讯」「新增 20 个候选选题」「1 篇待终审」。过程日志、模型调用、原始错误默认折叠。

### 2.2 待办驱动

首页第一屏必须回答：

- 今天是否跑过？
- 当前跑到哪一步？
- 有几篇文章需要我审？
- 有几篇文章卡住？
- 是否需要我点击重试或重建？

### 2.3 普通视图与调试视图隔离

普通用户不直接看到：

- AI 提示词全文
- 模型原始回复
- 内部 error stack
- 来源补全内部备注

调试信息可以在「高级信息」「AI 调用记录」中折叠展示，并通过权限控制。

### 2.4 状态颜色统一

| 颜色 | 含义 | 用例 |
|---|---|---|
| 绿色 | 成功、通过、可发布 | 运行成功、事实核查通过、批准发布 |
| 黄色 | 警告、待处理、需要补充 | 有源失败、待补来源、待终审 |
| 红色 | 失败、拒绝、未通过 | 文章生成失败、事实核查未过、打回 |
| 蓝色 | 进行中 | 当前运行步骤、后台执行中 |
| 灰色 | 跳过、归档、已被替代 | 未就绪跳过评分、历史 run 被替代 |

## 3. 信息架构

建议主导航为 5 个一级入口。

| 页面 | 目的 | 主要用户 |
|---|---|---|
| 今日看板 | 查看当天流水线、待办和运行控制 | 内容运营 |
| 文章库 | 管理所有文章，按状态筛选 | 内容运营 |
| 运行历史 | 查看每次流水线执行结果与失败原因 | 运营负责人、技术 |
| 选题池 | 理解今天为什么选这些题，哪些题被拒 | 内容运营 |
| 配置管理 | 管理关键词、采集源、策略文档 | 运营负责人、技术 |

文章详情页从文章库、今日看板和运行详情进入，不作为一级导航。

## 4. 首页 / 今日看板

### 4.1 页面目标

今日看板是默认首页。它要用一屏说明当天生产情况，并把用户需要处理的事项放在最醒目的位置。

### 4.2 首屏模块

#### A. 顶部状态栏

展示：

- 日期：今日 daily key，例如 `2026-06-05`
- 今日运行状态：未开始 / 运行中 / 成功 / 部分成功 / 失败
- 开始时间、结束时间、耗时
- 今日产出摘要：资讯数、候选选题数、生成文章数、待终审数

主按钮：

| 按钮 | 出现条件 | 行为 |
|---|---|---|
| 跑今天 | 当天没有 active daily run | 触发每日运行 |
| 重试失败 | 今日 run 为失败或部分成功 | 只补失败步骤 |
| 重建今天 | 今日已有 run | 二次确认后归档旧数据并完整重跑 |

按钮状态要求：

- 不可点击时必须说明原因，例如「今天已运行，不能重复开工」。
- 重建今天必须使用高风险确认弹窗，说明旧数据会被归档但不会物理删除。
- 所有触发动作无论成功或被拒，都应在操作记录中可查。

#### B. 待我处理

这是首页最重要模块，位置应高于流水线细节。

展示卡片：

- 待终审文章 N 篇
- 待补来源文章 N 篇
- 运行失败/部分失败 N 项
- 采集源持续失败 N 个

每张卡片点击后进入对应筛选结果：

- 待终审 → 文章库 `status=ready_for_review`
- 待补来源 → 文章库 `status=needs_fact_sources`
- 运行失败 → 今日运行详情
- 源失败 → 今日运行来源明细

#### C. 7 步流水线

使用横向步骤条或纵向时间线展示当天 pipeline。

| 步骤 | UI 主标题 | 结果摘要示例 | 详情入口 |
|---|---|---|---|
| 1 | 采集选题源 | 抓到 222 条，入库 180 条，3 个源失败 | 查看采集源 |
| 2 | 生成主题池 | 新增 20 个候选，拒绝 1 个重复题 | 查看选题池 |
| 3 | 创建文章任务 | 选中 1 个 92 分 P0 选题 | 查看任务 |
| 4 | 生成文章 | 生成 1 篇，质量分 86 | 查看文章 |
| 5 | 事实核查 | 微调后可发 / 待补来源 3 条 | 查看核查 |
| 6 | 渠道改写 | 公众号、抖音、小红书 3/3 成功 | 查看渠道稿 |
| 7 | SEO/GEO 评分 | SEO 82 / GEO 78 / 综合 80 | 查看评分 |

每个步骤卡片应包含：

- 状态徽章
- 核心数字
- 一条人能读懂的结果摘要
- 展开后的详细数据
- 出错时的简短错误原因和「查看详情」

#### D. 最近文章

展示今日生成或进入终审的文章列表，最多 5 条。

字段：

- 标题
- 状态
- 质量分
- SEO/GEO 分
- 事实核查结论
- 主操作：查看 / 终审

## 5. 文章库

### 5.1 页面目标

文章库用于查找、筛选和进入文章详情。设计重点是状态筛选和终审效率。

### 5.2 筛选器

必须支持：

- 状态筛选：全部、待补来源、待终审、已复审、批准发布、已发布、已打回、已归档
- 时间筛选：今天、近 7 天、近 30 天、自定义
- 分数筛选：质量分、SEO 分、GEO 分
- 搜索：标题、slug、关键词

### 5.3 列表字段

| 字段 | 说明 |
|---|---|
| 标题 | 文章标题，点击进入详情 |
| 状态 | 使用统一状态徽章 |
| 质量分 | 0-100 |
| 事实状态 | 通过 / 待补来源 / 未过 |
| SEO/GEO | 双分数并列展示 |
| 渠道稿 | 公众号/抖音/小红书完成情况 |
| 更新时间 | 最近状态更新时间 |
| 操作 | 查看、终审、打回原因 |

### 5.4 空状态

| 场景 | 文案方向 |
|---|---|
| 今日无文章 | 今天还没有生成文章，可先运行今日流水线 |
| 无待终审 | 当前没有需要人工处理的文章 |
| 筛选无结果 | 没有符合当前筛选条件的文章 |

---

## 6. 文章详情

### 6.1 页面目标

文章详情是人工终审的核心页面。设计目标是让运营可以高效阅读正文、理解机器判断、查看渠道稿，并完成通过或打回。

### 6.2 页面布局

建议采用左右结构：

- 左侧主区域：文章正文 HTML 渲染
- 右侧信息区或顶部 tabs：版本、质量、事实核查、来源补全、渠道稿、SEO/GEO、历史、AI 调用记录

移动端或窄屏可以改为上下结构。

### 6.3 顶部摘要区

展示：

- 文章标题
- 当前状态进度条
- 质量分
- 事实核查结论
- SEO/GEO 分
- 版本号，例如 v1、v2、v3
- 字数
- 生成时间

状态机展示：

```text
已过质量门 → 待补来源 → 待终审 → 已复审 → 批准发布 → 已发布
                         ↘ 不通过/归档
```

当前状态高亮，历史状态置灰，异常分支用红色或灰色。

### 6.4 终审操作

当文章状态为 `ready_for_review` 时，展示固定操作区：

| 操作 | 要求 |
|---|---|
| 通过 | 将文章标记为已复审或批准发布，按实际权限拆分 |
| 打回 | 必须填写原因 |
| 查看渠道稿 | 快速跳到渠道稿 tab |

打回弹窗应包含：

- 打回原因输入框
- 可选原因快捷项：事实不充分、表达不合适、选题不适合、结构问题、其他
- 确认后进入 `rejected`，写入审计记录

### 6.5 Tabs 信息架构

#### 正文

展示最新版本正文，保留标题、段落、表格、FAQ 和来源链接的可读性。

需要支持版本切换：

- v1 初稿
- v2/v3 来源修订版
- 当前最新版本标记

#### 质量报告

展示：

- 总分
- 7 个维度明细
- publish / revise / reject 结论
- 问题清单

不要展示模型原始 JSON，除非进入 Debug 权限视图。

#### 事实核查

展示每轮核查：

- 表述总数
- 高风险条数
- 必修条数
- 发布就绪结论
- 待修清单

需要突出「用户是否要介入」：

- 可自动修：提示系统正在补来源
- 连续修不动：提示需要人工处理

#### 来源补全

按待修表述展示：

- 原表述
- 找到的权威来源
- 建议改写
- 当前处理状态

内部 notes 默认不展示。

#### 渠道稿

三个渠道独立展示：

- 公众号长文
- 抖音口播稿
- 小红书笔记

每个渠道显示：

- 标题
- 状态
- 字数
- 正文
- 复制按钮

渠道失败时只影响对应渠道，不应让用户误以为整篇文章失败。

#### SEO/GEO 评分

展示：

- SEO 总分
- GEO 总分
- 综合分
- SEO 9 项明细
- GEO 8 项明细
- 关键扣分项

若文章未就绪而跳过评分，显示灰色跳过状态和原因。

#### 历史

展示状态变化时间线：

- 时间
- 操作人或系统
- 从什么状态到什么状态
- 原因

#### AI 调用记录

默认折叠，仅调试或管理员可见。

展示摘要：

- 任务类型
- 模型名
- 状态
- 耗时
- prompt 字符数
- response 字符数
- 错误摘要

默认不展示 prompt 全文和 raw response。

## 7. 运行历史

### 7.1 页面目标

运行历史用于查看每次流水线执行是否稳定、是否失败、是否被重建替代。

### 7.2 列表字段

| 字段 | 说明 |
|---|---|
| 日期 / daily key | 每日运行日期 |
| 状态 | 成功、部分成功、失败、运行中、已被替代 |
| 模式 | start、retry、rebuild、force |
| 触发来源 | Web、CLI、Viewer |
| 触发人 | actor |
| 产出 | 选题数、文章数、核查数、渠道稿数 |
| 耗时 | started_at 到 finished_at |
| 操作 | 查看详情、重试失败、查看操作记录 |

### 7.3 运行详情

运行详情包含：

- 7 步流水线执行结果
- 每步耗时
- output summary
- warning summary
- 错误摘要
- 关联文章
- 操作记录
- 状态变更记录
- 最近事件流
- 失败模型调用摘要

技术细节默认折叠，错误首行摘要直接展示。

### 7.4 采集源明细

以表格展示每个源：

- source name
- source group
- source type
- URL 或 query
- 状态
- HTTP 状态
- 抓到条数
- 入库条数
- 耗时
- 失败原因
- sample titles

需要支持筛选：

- 状态：成功、部分成功、失败、跳过
- source group
- HTTP status

设计上要突出「持续失败的源」，为未来维护源质量提供入口。

## 8. 选题池

### 8.1 页面目标

选题池用于解释「今天为什么写这篇」，让运营理解候选选题、评分和拒绝原因。

### 8.2 列表字段

| 字段 | 说明 |
|---|---|
| 标题 | 候选选题标题 |
| 分数 | 0-100 |
| 优先级 | P0/P1/P2 |
| 状态 | 候选、已选中、已生成、被拒 |
| 拒绝原因 | 与最近文章重复、分数不足等 |
| 来源摘要 | 可展示来源数量或代表标题 |
| 创建时间 | 选题生成时间 |

### 8.3 设计重点

- 高分题应容易扫描。
- 被拒题不要隐藏，运营需要知道系统为什么没选。
- 已选中题应能跳转到对应文章。

## 9. 配置管理

### 9.1 页面目标

配置管理是未来 Web 版的核心能力，用于在线维护关键词、采集源和策略文档。当前 PRD 只要求设计入口和基础信息架构，不要求展开复杂编辑器细节。

### 9.2 子页面

| 子页面 | 功能 |
|---|---|
| 关键词库 | 查看、启用/停用、增删改关键词 |
| 采集源 | 查看 42 个源，启用/停用，维护 URL/类型/分组 |
| 策略文档 | prompts、schemas、production policy 等配置版本 |

### 9.3 配置编辑要求

- 修改策略文档时要显示版本号、更新时间、更新人。
- 高风险配置要支持版本历史和回滚。
- prompt/schema 修改会影响生成质量，保存前应有确认提示。
- 配置修改在下一次 engine run 生效，不要求实时影响当前运行。

## 10. 关键流程

### 10.1 跑今天

触发条件：

- 今日没有 active daily run。

用户路径：

1. 用户进入今日看板。
2. 点击「跑今天」。
3. 系统接受请求并显示运行中。
4. 流水线步骤逐步更新。
5. 完成后展示成功、部分成功或失败。

异常：

- 如果今日已运行，按钮禁用或请求被拒，显示原因。
- 被拒操作也要记录在操作历史中。

### 10.2 重试失败

触发条件：

- 今日 run 状态为失败或部分成功。

设计要求：

- 文案说明「只补失败步骤，不重复已成功步骤」。
- 重试后仍归属今日 daily key。
- 运行详情中能看到 retry 操作记录。

### 10.3 重建今天

触发条件：

- 今日已有 run。

设计要求：

- 必须二次确认。
- 文案说明旧数据归档，不物理删除。
- 已批准发布或已发布的文章不自动归档，需要 warning。
- 新 run 应成为 active run，旧 run 显示已被替代。

### 10.4 人工终审

触发条件：

- 文章状态为 `ready_for_review`。

用户路径：

1. 用户从首页待办进入待终审列表。
2. 打开文章详情。
3. 阅读正文、质量报告、事实核查和渠道稿。
4. 点击通过或打回。
5. 系统写入 review action 和 status transition。

打回必须填写原因。

### 10.5 待补来源修订

触发条件：

- 事实核查结果为待补来源。

设计要求：

- 默认提示系统会自动处理。
- 展示修订轮次 v1 → v2 → v3。
- 当连续修不动时，突出显示需要人工介入的待修清单。

## 12. 数据口径附录

正式 Web 项目直接读取 MySQL，不依赖本仓库本地 Viewer。Viewer 只作为调试参考。

### 12.1 前端主要读取表

| 场景 | 表 |
|---|---|
| 文章列表与详情 | articles、article_versions |
| 渠道稿 | channel_outputs |
| 发布包 | publish_packages |
| 质量与核查 | quality_reports、fact_checks、source_resolutions |
| SEO/GEO | seo_geo_scores |
| 人工审核 | review_actions、status_transitions |
| 运行监控 | engine_runs、workflow_steps、workflow_events、source_collection_logs |
| AI 调用摘要 | model_runs |
| 配置管理 | config_keywords、config_sources、app_configs |

### 12.2 适合普通 UI 展示的字段

- articles：title、slug、status、quality_score、seo_score、geo_score、fact_publish_readiness

- channel_outputs：channel、title、content_markdown、status
- workflow_steps：step_name、status、duration_ms、output_summary_json
- source_collection_logs：source_name、source_group、status、http_status、items_found、items_inserted、duration_ms、error_message
- status_transitions：from_status、to_status、reason、actor、created_at
- model_runs：task_type、status、duration、error_message 摘要

- model_runs.raw_response
- 内部 error stack

## 14. 验收标准

UI 设计通过验收需满足：

1. 首页第一屏能清楚回答「今天是否跑过、跑到哪、产出多少、哪些要我处理」。
2. 7 步流水线能同时表达状态、结果数字和异常入口。
3. 待终审文章从首页到详情再到通过/打回，路径不超过 3 次主要点击。
4. 文章详情能支持正文阅读、版本切换、质量/事实/渠道/评分查看。
5. 待补来源文章能看清自动修订轮次和需要人工介入的待修清单。
6. 运行失败时，普通用户能看到简短原因，技术用户能展开查看详细 trace。
7. 状态颜色和状态文案在所有页面保持一致。
8. 普通用户默认看不到 prompt、模型原始回复和内部错误堆栈。
9. 重试失败和重建今天的风险差异在 UI 上明确可见。
10. 空状态、失败态、跳过态均有明确文案，不出现只有空白表格的页面。
